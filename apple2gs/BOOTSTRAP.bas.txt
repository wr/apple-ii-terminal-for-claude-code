Serial bootstrap loader v3 for the IIgs SHR client (claude.obj / COBJ).

Self-contained installer: dials the bridge if the modem isn't connected,
receives the client binary into $4000, verifies it against a checksum sent
over the wire, and BSAVEs it to disk. NOTHING in this listing is tied to a
specific build — length and checksum arrive in a 5-byte preamble from the
bridge (sum lo/mid/hi, len lo/hi, stored at $2F9-$2FD), so this program never
needs to change when the client is rebuilt. Type it once, SAVE LOADER, done.

Updating the client forever after:
  Mac:  ./build.sh, run bridge with --bootstrap ../apple2gs/COBJ
        (the file is re-read on every transfer - rebuilds are picked up live)
  IIgs: boot DOS from the FloppyEmu, RUN LOADER
That's the whole procedure. The loader auto-dials, auto-verifies, auto-BSAVEs.

Protocol (bridge side: bridge.py run_bootstrap):
  loader sends 'R'         -> bridge streams preamble + binary at ~400 B/s
  (any CR the loader sends -> bridge answers 0x06; harmless, drained)
The loader ALWAYS dials - a "probe first" scheme can't work because the modem
echoes the probe byte in command mode, indistinguishable from an answer. When
already online, the dial string just reaches the bridge's wait loop, which
ignores everything but 'R'.
The receive loop is 78 bytes of ML at $300 (polls every ~20 us; the Applesoft
loop it replaced dropped bytes whenever WiFi bursts beat the 3-byte SCC FIFO).
Uses zero page $EB-$EE; safe under Applesoft+DOS.

Lines 10-40 program the SCC (channel B = modem port) for 9600 8N1, polled.
A real IIgs never initializes the SCC at power-on - Control Panel settings
alone do NOT make the port live (Apple IIgs TN #018). Same init claude.s does.

10 SB=49208:DB=49210:S=PEEK(SB)
20 FOR J=1 TO 12:READ R,V:POKE SB,R:POKE SB,V:NEXT
30 DATA 9,64,4,68,3,192,5,234,11,80,12,10
35 DATA 13,0,14,1,15,0,1,0,9,0,3,193
40 FOR J=1 TO 4:S=PEEK(DB):NEXT
50 PRINT "DIALING...":D$="ATDS=0":FOR J=1 TO LEN(D$):POKE DB,ASC(MID$(D$,J,1)):NEXT:POKE DB,13
52 S=PEEK(SB):IF INT(S/2)*2=S THEN 52
54 FOR J=1 TO 2000:S=PEEK(SB):IF INT(S/2)*2<>S THEN S=PEEK(DB):J=0
56 NEXT
60 FOR J=1 TO 4:S=PEEK(DB):NEXT
62 C=0:FOR J=0 TO 77:READ V:POKE 768+J,V:C=C+V:NEXT
64 IF C<>11941 THEN PRINT "ML TYPO, SUM ";C;" WANT 11941":END
66 DATA 160,0,169,0,133,235,169,64,133,236,169,82
67 DATA 141,58,192,162,0,173,56,192,41,1,240,249
68 DATA 173,58,192,157,249,2,232,224,5,208,238,173
69 DATA 252,2,133,237,173,253,2,133,238,173,56,192
70 DATA 41,1,240,249,173,58,192,145,235,230,235,208
71 DATA 2,230,236,165,237,208,2,198,238,198,237,165
72 DATA 237,5,238,208,224,96
74 PRINT "RECEIVING...":CALL 768
80 N=PEEK(764)+256*PEEK(765):A=16384
82 W=PEEK(761)+256*PEEK(762)+65536*PEEK(763)
84 C=0:FOR I=0 TO N-1:C=C+PEEK(A+I):NEXT
86 IF C<>W THEN PRINT "BAD SUM ";C;" WANT ";W;" - RUN TO RETRY":END
88 PRINT CHR$(4);"BSAVE COBJ,A$4000,L";N
90 PRINT "INSTALLED ";N;" BYTES"

Line by line:
  10-40  SCC init (see above) + drain stale bytes from the 3-byte FIFO
  50-56  dial phone book entry 0 (always - see protocol note above), wait
         for the echo/CONNECT/bridge-ack chatter to go quiet before moving on
  60     drain leftovers
  62-72  install the ML receive loop at $300, self-checked against 11941 -
         a mistyped DATA value stops here instead of running garbage
  74     ML sends 'R', reads preamble + binary (~10s at 400 B/s)
  80-86  checksum the received bytes against the transmitted sum (~30s)
  88     BSAVE with the transmitted length - the install step
Client changes that move load address or exceed ~20KB need line 80's A and the
ML's $40 (line 66, 8th value) updated; nothing else ever changes.

If it freezes at RECEIVING: bridge isn't in --bootstrap mode or the connection
is down; the ML loop can't be Ctrl-C'd - Ctrl-Reset gets you back, program
intact, RUN to retry. BAD SUM: RUN again, the bridge re-streams on every 'R'.
